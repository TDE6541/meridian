# Meridian Team Charter

## Purpose

Keep Meridian work governed, legible, and aligned to repo truth.

## Mission

Advance Meridian as a governed city digital twin intelligence repo through narrow, evidence-backed changes.

## Project Identity

Meridian is a governed city digital twin intelligence effort with root canon, entity/schema discipline, a transport-only Wave 3 bridge substrate, a bounded Wave 4A governance runtime lane, a bounded Wave 4B meeting-capture lane with a local/frozen governance handoff seam, a Wave 4.5 calibration truth lane, a bounded Wave 5 authority-topology lane, a bounded Wave 6 forensic-chain lane, a bounded Wave 7 civic skins rendering lane, a Wave 8 corridor scenario integration lane, a Wave 9 local dashboard proof lane, a V2-A local/demo-day live civic nervous system extension, a V2-B/GARP Authority Runway local sublane, and a V2-B Foreman/Auth proof cockpit with AUTH-5 deployed demo proof recorded. Meridian V1 is complete through Wave 9. Wave 9 is the final V1 wave, and there is no Wave 10 in V1.

## Role Definitions

- Architect: sets scope, defines canon, and approves execution boundaries.
- Execution lane: performs the approved work with minimal diffs and explicit verification.
- Reviewer: checks output against repo truth, acceptance criteria, and closeout requirements.

## Relay Discipline

Meridian uses a five-room relay structure.

1. Intake: capture the request, scope, and governing inputs.
2. Canon: confirm truth sources, posture, and non-negotiables.
3. Plan: define the minimal change set and verification path.
4. Execution: perform only the approved work.
5. Closeout: verify results, surface HOLDs, and prepare signoff.

## Governance Principles

- HOLD > GUESS
- Evidence first
- Minimal diffs
- Clear contracts
- Explicit verification
- No silent scope growth

## Closeout Standard

Every session ends with changes made, acceptance status, remaining HOLDs, contract and migration status, blast radius, exact files touched, and signoff state.
