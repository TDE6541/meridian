# Meridian Team Charter

## Purpose

Keep Meridian work governed, legible, and aligned to repo truth.

## Mission

Advance Meridian as a governed city digital twin intelligence repo through narrow, evidence-backed changes.

## Project Identity

Meridian is a governed city digital twin intelligence effort with root canon, entity/schema discipline, a transport-only Wave 3 bridge substrate, and a bounded Wave 4A governance runtime landing zone with a static civic policy pack and Block C runtime subset integration.

## Role Definitions

- Architect: sets scope, defines canon, and approves execution boundaries.
- Execution lane: performs the approved work with minimal diffs and explicit verification.
- Reviewer: checks output against repo truth, acceptance criteria, and closeout requirements.

## Relay Discipline

Meridian uses a five-room relay structure.

1. Intake: capture the request, scope, and governing inputs.
2. Canon: confirm truth sources, posture, and non-negotiables.
3. Plan: define the minimal change set and verification path.
4. Execution: perform only the approved work.
5. Closeout: verify results, surface HOLDs, and prepare signoff.

## Governance Principles

- HOLD > GUESS
- Evidence first
- Minimal diffs
- Clear contracts
- Explicit verification
- No silent scope growth

## Scope Discipline

Wave 1 remains the repo-native foundation. Wave 2 remains the bounded entity ontology extension. Wave 3 adds only a transport-only NATS bridge substrate, a fail-closed governance transport stub, fixture-backed proof, and the required canon sync. Wave 4A adds only a bounded `command_request` governance runtime landing zone, adapter activation, the static local civic policy pack, and Block C runtime subset integration that may emit `ALLOW`, `SUPERVISE`, `HOLD`, and `BLOCK`. Wave 4A does not ship event-side routing, publisher widening beyond the existing `HOLD` path, actor-level authority topology, entity mutation, KV mutation, or civic ForensicChain runtime writes.

## Closeout Standard

Every session ends with changes made, acceptance status, remaining HOLDs, contract and migration status, blast radius, exact files touched, and signoff state.
